# Amber 2.0 — Working Spec

Working document for 2.0 planning. Goal: identify shared infrastructure across features, surface what the codebase already provides vs what needs building, define build order to avoid rework.

## Current state — what exists

### RHI rendering pipeline

**Allocation pattern:** Per-frame transient resources, no caching. `rhi_blit()` (`renderfunctions.cpp:62-175`) is the main hotspot — each call creates 1 pipeline + 1 SRB + 1 UBO buffer, all destroyed at end of frame via `qDeleteAll(params.transientResources)`. A single clip with 3 effects = ~10 pipelines + ~10 SRBs + ~15 buffers created and destroyed per frame.

**What IS cached:** Clip-level textures persist across frames — ping-pong RGBA8 textures (`ClipRhiResources` in `renderfunctions.h:34-37`), YUV plane textures (`clip.h:172-178`), and effect superimpose textures (`effect.h:260`). These are allocated once per clip and reused until `Clip::Close()`.

**Global resources:** Single `QRhi` instance (`renderthread.cpp:98-150`), one fullscreen quad VBO, one vertex UBO, one sampler, 5 core `.qsb` shaders, 3 main compositing textures. All persist for app lifetime.

**Deferred deletion:** `RenderThread::DeferRhiResourceDeletion()` provides thread-safe cleanup — main thread queues, render thread drains once per frame after `endOffscreenFrame()`.

### Effect system

**14 internal effects** registered via `EffectInternal` enum (`effect.h:78-95`): Transform, Text, Solid, Noise, Volume, Pan, Tone, Shake, Timecode, Mask (disabled), FillLeftRight, VST, CornerPin, Frei0r, RichText.

**Factory:** `Effect::Create()` (`effect.cpp:96-143`) — switch on enum, returns `shared_ptr<SpecificEffect>`. External XML effects fall through to generic `Effect` base class.

**Registration:** `load_internal_effects()` (`effectloaders.cpp:40-135`) — populate `EffectMeta` (name, category, internal enum, type/subtype), append to global `effects` vector.

**Flags:** `ShaderFlag` (GPU pipeline), `CoordsFlag` (transform matrix), `SuperimposeFlag` (CPU QPainter → GPU upload), `ImageFlag` (CPU pixel buffer, forces RGBA decode path).

**Adding a new internal effect requires:** enum value in `effect.h`, factory case in `effect.cpp`, loader entry in `effectloaders.cpp`, header+impl files in `effects/internal/`. ~20 lines of boilerplate total.

### Generator clip creation

`CreateObjects` enum (`timeline.h:39-46`): ADD_OBJ_TITLE, ADD_OBJ_SOLID, ADD_OBJ_BARS, ADD_OBJ_TONE, ADD_OBJ_NOISE, ADD_OBJ_AUDIO. Menu in `Timeline::add_btn_click()` (`timeline.cpp:1131-1169`). Clip instantiation in `TimelineWidget::mouseReleaseCreating()` (`timelinewidget.cpp:796-825`) — creates `Clip(sequence)`, `set_media(nullptr)`, attaches default Transform + the generator effect.

### Text rendering (SuperimposeFlag)

**TextEffect** (`texteffect.cpp`, ~300 lines): QPainterPath-based. Full shadow (angle/distance/blur/opacity), outline (stroke width/color), word wrap, 4-way alignment, position offset. Renders into `QImage img` (inherited from Effect base). `process_superimpose()` in the base class handles GPU upload — subclasses only override `redraw(timecode)`.

**RichTextEffect** (`richtexteffect.cpp`, ~190 lines): QTextDocument + HTML. Shadow via composition mode tinting. No outline support. Autoscroll feature.

**Change detection:** `valueHasChanged(timecode)` (`effect.cpp:1000-1026`) — caches all field QVariants, compares on each frame. Redraw only happens when a parameter actually changes. Efficient for static text.

### Serialization (.ove)

**Fields matched by string ID**, not position — safe for reordering and evolution. `Effect::save()` writes `<row>/<field>` with id/value/keyframes. `Effect::load()` reads them back.

**Custom data hook:** `virtual custom_load(QXmlStreamReader&)` — called for unknown child elements inside an effect's XML block. VSTHost uses this for `<plugindata>` blobs. SubtitleEffect can use it for `<subtitles>/<entry>` elements.

**Unknown effect handling:** `VoidEffect` wraps unknown effects, preserves their XML verbatim for round-trip. Projects saved with 2.0 effects can be opened in 1.x without data loss (effect appears as disabled/void).

### Window patterns

**Docked panels:** `Panel` extends `QDockWidget` (`ui/panel.h`). Auto-registers in `amber::panels` vector. Layout save/restore via `LoadLayoutState()`/`SaveLayoutState()`. 7 panels: Project, EffectControls, SequenceViewer, FootageViewer, Timeline, GraphEditor, UndoHistory.

**Modeless floating:** `DebugDialog` pattern — created once in MainWindow constructor, stored as singleton, toggled via menu. Non-blocking.

**Modal blocking:** Most dialogs in `dialogs/` — `ExportDialog`, `PreferencesDialog`, `TextEditDialog`, etc. Created on-demand, `exec()`, destroyed after.

**Dialog→timeline communication:** Direct access to global panel singletons (`panel_timeline`, `panel_effect_controls`, etc.) via `#include "panels/panels.h"`. No signal infrastructure for playhead tracking — dialogs that need it would need to connect to viewer signals.

### Viewer display pipeline

**Round-trip:** RenderThread composites to GPU texture (`front_tex_[N]`) → `readBackTexture()` copies full-res RGBA to CPU (`cpu_frame_[N]` QByteArray) → `endOffscreenFrame()` blocks until done → ViewerWidget::render() locks mutex, uploads CPU buffer to a new GPU texture → draws textured quad via passthrough shader.

**Double-buffered:** Two CPU buffers + two GPU textures, ping-pong via `front_buffer_switcher` atomic. Thread-safe but not async — `endOffscreenFrame()` blocks until readback completes.

**Readback is always full-res.** No half-res path exists. 1080p = 8.3 MB/frame. 4K = 35 MB/frame. At 30fps 4K = 1.05 GB/s bandwidth.

**No frame skipping during scrub.** The `scrubbing` flag only affects decode timeout (250ms vs 500ms in cacher), not readback frequency. Fast scrubbing still reads back every frame.

**Fullscreen copies the entire frame.** `ViewerWindow::set_frame()` does a full `memcpy` of the readback buffer (`viewerwindow.cpp:82`). If fullscreen is visible during playback, that's 2× bandwidth.

### Viewer overlays

**Architecture:** `ViewerOverlay` is a **sibling QWidget** of `ViewerWidget`, not a child — both are children of `ViewerContainer`. This is a workaround for Vulkan child QWidget compositing failures.

**Drawing:** `ViewerOverlay::paintEvent()` (`viewerwidget.cpp:1058-1071`) delegates to ViewerWidget methods via QPainter:
- `draw_title_safe_area()` — action safe (90%) + title safe (80%) rectangles + center crosshair
- `draw_guides()` — user-placed horizontal/vertical guide lines, zoom-aware
- `draw_gizmos()` — effect transform handles (DOT, POLY, TARGET types)

**Coordinate transform:** Image-space → widget-space via `container->zoom` scale factor + scroll offsets. Gizmos use inverted Y (render convention Y=0 at bottom vs Qt Y=0 at top).

**Mouse interaction:** Guide dragging, gizmo selection/movement, hover cursors — all in ViewerWidget event handlers, with 5px hit-test threshold.

### Audio pipeline

**Threading:** 3 threads access shared audio state:
- **UI/Main thread** — calls `clear_audio_ibuffer()` on seek/pause, locks both `audio_thread->lock` and `audio_write_lock`
- **AudioSenderThread** (TimeCriticalPriority) — reads ring buffer, writes to output device
- **Cacher threads** (one per clip) — decode + mix into ring buffer, lock `audio_write_lock`

**Shared state (current synchronization):**
- `audio_ibuffer_frame` — `std::atomic<long>` ✓
- `audio_ibuffer_read` — `std::atomic<qint64>`, uses `fetch_add()` ✓
- `audio_reset_` — `std::atomic<bool>` per Cacher ✓
- `audio_scrub_id`, `audio_scrub_data_ready` — `std::atomic` ✓

**NOTE: ROADMAP is outdated.** It says these are "plain types, no atomics, no locks" — they've been upgraded to `std::atomic<>` since it was written.

**Remaining data race:** `audio_ibuffer[192000]` (the ring buffer itself) is **not protected** for AudioSenderThread reads. Cacher writes are guarded by `audio_write_lock` (`cacher_audio.cpp:447-448`), but AudioSenderThread reads/clears at `audio.cpp:234, 247, 272` have **no lock**. Torn reads possible.

**Scrub grain:** 80ms windowed audio + Hann fade. `staging_buffer_` heap-allocated per grain (`audio.cpp:222`) — should be pre-allocated. No debouncing for rapid scrubs (<16ms between seeks).

### YUV upload path

**Copy chain:** `Clip::Retrieve()` (`clip.cpp:594-626`) copies each AVFrame plane into a `QByteArray` before `uploadTexture()`. For 1080p YUV420P: Y=2.0 MB + U=0.5 MB + V=0.5 MB = **3.0 MB per frame**.

**Zero-copy verdict: not feasible.** `QRhiTextureSubresourceUploadDescription` only accepts `QByteArray` — no raw `const void*` constructor in Qt 6.8+ RHI. The copy is mandatory.

**Practical impact: negligible.** 90 MB/s at 30fps is 0.45% of CPU memory bandwidth. Decode/filter (5-50ms) and GPU shader conversion (1ms) dominate. The 3 MB copy completes in ~33µs.

**Optimization worth doing:** Pre-allocate persistent QByteArrays instead of constructing new ones per frame. Eliminates heap allocation churn, not the copy itself.

## Shared infrastructure

Cross-cutting work that benefits multiple features.

### SI-1: RHI resource caching

**Consumers:** Every shader effect (existing ~29 + new 15-20 GPU-native + ShaderToy), scopes, viewer overlays.

**Current cost:** `rhi_blit()` creates pipeline+SRB+buffer per call. Pipeline creation is 100µs+ on Vulkan. With 2.0 adding more shader effects per clip, this scales linearly worse.

**Approach:** Cache by (shader pair + vertex layout + blend mode) key. `QRhiGraphicsPipeline` is the expensive one — SRBs and buffers are cheap but noisy. A `RhiPipelineCache` keyed on shader+blend config would eliminate 80%+ of per-frame allocations. Buffer pooling (single large buffer with offset allocation) handles the rest.

**Integration point:** `rhi_blit()` in `renderfunctions.cpp:62-175` — the single function where all transient resources are created. Contained change.

**Risk:** QRhi pipeline objects may hold state that makes sharing unsafe across different render pass instances. Needs testing.

### SI-2: Floating tool window

**Consumers:** Subtitle editor, color correction tool.

**Existing pattern to follow:** `DebugDialog` — persistent modeless QDialog, singleton, toggled via menu.

**What's missing vs DebugDialog:** Bidirectional timeline sync. No existing signal infrastructure for "playhead moved" → "dialog updates". Need to connect to `ViewerWidget` or `Sequence` playhead signals. The reverse direction (dialog click → seek) is easy — call `panel_sequence_viewer->seek(frame)`.

**Decision deferred:** Base class vs repeated pattern. Build subtitle editor first, then color correction. Extract if they converge. KISS.

### SI-3: Superimpose text utilities — MAYBE

**Consumers:** SubtitleEffect, Timer/Countdown, Progress bar.

**TextEffect's `redraw()` breakdown:**
- Word wrap algorithm (lines 183-203) — reusable but simple enough to copy
- Shadow rendering: QPainterPath copy + translate + fill + blur (lines 263-285) — ~25 lines
- Outline rendering: QPen stroke on path (lines 288-295) — ~10 lines
- Text positioning/alignment (lines 220-260) — tightly coupled to TextEffect's field layout

**Verdict:** Copy, don't extract. The shadow+outline code is <40 lines. The alignment logic is tied to each effect's specific field setup. Extracting a utility would create an abstraction that's harder to modify than the copy. TextEffect stays untouched. Timer/Progress bar may not even need text shadow/outline.

### SI-4: Multi-pass shader support

**Consumers:** 3-way color corrector (lift/gamma/gain), denoise, ShaderToy Buffer A/B/C/D (post-2.0).

**Current state:** `process_shader()` (`effect.cpp`) runs one fragment shader per effect iteration. Ping-pong textures (`ClipRhiResources`) alternate input/output. Multi-pass = multiple iterations with different shader params, which the iteration system already supports for blur.

**What's actually new:** Most "multi-pass" needs are already handled by the iteration count mechanism. True multi-pass (different shaders per pass with intermediate state) is only needed for ShaderToy buffers — which is post-2.0 scope. Lift/gamma/gain can be a single shader with 9 uniform params (3 RGB triplets).

**Revised verdict:** Less work than initially thought. Iteration mechanism covers denoise (box blur passes). LGG is single-pass. SI-4 may not need to exist as a separate infrastructure item for 2.0.

## Feature specs

### F-1: GPU-native effects + Frei0r deprecation

**Depends on:** SI-1 (RHI caching — nice-to-have, not blocking)

Shaders slot into existing XML+GLSL pipeline in `effects/shaders/`. Each shader = `.xml` descriptor + `.frag` file. Auto-discovered at startup by `effectloaders.cpp`. No C++ code needed per shader.

**GPU-only pipeline = disabling Frei0r.** The goal isn't eliminating the CPU path from the code, it's making Frei0r opt-in. Once common Frei0r effects have GPU shader equivalents, Frei0r becomes a legacy toggle:
- Runtime setting: "Enable Frei0r legacy effects (CPU, slow)" — off by default
- Compile flag: `-DNOFREI0R` already exists, strips the bridge entirely
- With Frei0r off, `NeedsCpuRgba()` is never true → pure GPU pipeline for all clips
- Temporal Frei0r effects (deshake, stabilization) have no GPU equivalent — users who need these re-enable the toggle

**Already works without SI-1** — just less efficient. Can land shaders first, optimize later.

### F-2: ShaderToy import

**Depends on:** Nothing. The runtime already does everything.

The existing shader pipeline handles it almost entirely:
- `QShaderBaker` with disk cache (`~/.cache/amber-editor/shaders/`) — already in `effect.cpp:695-738`
- UBO introspection via `QShaderDescription` — auto-discovers uniform offsets at load time
- `process_shader()` populates UBO by name matching (`entry.name == field->id()`) — field→uniform binding is automatic
- Bindings: 0=vertex UBO, 1=fragment UBO, 2-4=textures — rigid but sufficient

**What's actually new (~200-300 lines):**

A GLSL text preprocessor that transforms:
```glsl
void mainImage(out vec4 fragColor, in vec2 fragCoord) {
    vec2 uv = fragCoord / iResolution.xy;
    fragColor = texture(iChannel0, uv);
}
```
Into Amber's GLSL 440 format:
```glsl
#version 440
layout(std140, binding = 1) uniform FragParams {
    vec3 iResolution;
    float iTime;
    float iTimeDelta;
    int iFrame;
};
layout(binding = 2) uniform sampler2D iChannel0;
layout(location = 0) in vec2 vTexCoord;
layout(location = 0) out vec4 fragColor;

void mainImage(out vec4 fragColor, in vec2 fragCoord) {
    vec2 uv = fragCoord / iResolution.xy;
    fragColor = texture(iChannel0, uv);
}

void main() {
    vec2 fragCoord = vTexCoord * iResolution.xy;
    mainImage(fragColor, fragCoord);
}
```

That's it. The preprocessor injects the header (UBO declaration, sampler bindings, `main()` wrapper) and auto-generates an XML descriptor for effect discovery. `QShaderBaker` compiles it, `process_shader()` binds the uniforms. No new runtime code.

**Import UX:** Drop `.shadertoy` file into `effects/shaders/` (auto-discovered), or paste via *Effects → Import ShaderToy shader* dialog. Custom parameters via optional XML sidecar.

**2.0 scope:** Single-pass, single-input (`iChannel0`). Multi-pass (Buffer A/B/C/D), multi-channel, audio FFT — post-2.0.

### F-3: Subtitle effect

**Depends on:** SI-2 (floating window for bulk editing)

**Registration:** `EFFECT_INTERNAL_SUBTITLE` enum + factory case + loader entry. `ADD_OBJ_SUBTITLE` in `CreateObjects` enum + menu entry + clip creation logic. Generator clip: `set_media(nullptr)`, Transform + SubtitleEffect.

**Data model:** `QVector<SubtitleEntry>` (frame_in, frame_out, text) inside SubtitleEffect. NOT EffectFields — stored via `custom_load()`/`save()` override, same pattern as VSTHost's `<plugindata>`.

**Style fields:** Standard EffectRow/EffectField setup (font, size, color, alignment, outline, shadow) — copied from TextEffect constructor. These ARE EffectFields → keyframeable, serialized automatically.

**Rendering:** Override `redraw(timecode)`. Look up active subtitle entry by frame. If found, render text using QPainterPath (copied from TextEffect's shadow+outline+fill logic). If no active entry, fill transparent. `valueHasChanged()` handles redraw efficiently, but needs an override or `AlwaysUpdate()` since the active entry changes based on timecode, not field values.

**Import:** `.srt` parser → populate `QVector<SubtitleEntry>`. Triggered from File menu or floating editor. Creates generator clip + SubtitleEffect via ComboAction/UndoStack.

**Fine editing:** The text field in Effect Controls shows the active subtitle's text at playhead position. Modifying it updates the corresponding entry. Need a custom StringField behavior or a proxy that maps playhead → active entry → field value.

**Bulk editing (SI-2):** Floating QDialog with QTableWidget (columns: in timecode, out timecode, text). Double-click entry → seek. Import/export .srt from there.

**Serialization:**
```xml
<effect name="Render/Subtitle" enabled="1">
  <row><!-- style fields: font, size, color, etc. --></row>
  <subtitles>
    <entry in="100" out="200" text="First subtitle"/>
    <entry in="300" out="400" text="Second subtitle"/>
  </subtitles>
</effect>
```

Backward compatible: 1.x loads this, hits `VoidEffect`, preserves XML verbatim.

### F-4: Color correction tool

**Depends on:** SI-2 (floating window), F-6 (scopes), F-1 (LGG shader)

The shader is trivial (single-pass, 9 uniforms). The floating window with curves + wheels UI is the real work. Scopes provide live feedback.

**Open:** Curves vs wheels vs both. Separate design session needed.

### F-5: Timer / Countdown & Progress bar

**Depends on:** Nothing — standalone SuperimposeFlag effects.

Timer: `TimecodeEffect` already exists (`EFFECT_INTERNAL_TIMECODE`). New timer = similar but with configurable start/end/format and countdown mode. Progress bar: QPainter rectangle drawing in `redraw()`, configurable colors/thickness/position.

Low complexity, low dependency. Can land anytime.

### F-6: Scopes & monitoring

**Depends on:** SI-1 (reads GPU output, benefits from caching)

**Form factor:** Docked panels, not floating windows. Scopes in Resolve/Premiere/Kdenlive are all docked. Fits existing `Panel` system — auto-registers, layout save/restore works, toggleable from Window menu.

**Oak code reuse:** Needs audit. Oak scopes used raw `glReadPixels` + GL shaders. Amber needs `QRhiReadbackResult` + either CPU computation or QRhi compute shaders. CPU computation is simpler and fine for scope rendering (histogram = one pass over pixel buffer).

### F-7: 3-point editing

**Depends on:** Nothing shared. Much less work than expected.

**What already exists:**
- **Source monitor:** `panel_footage_viewer` — same `Viewer` class as sequence viewer, independent playback, fully functional
- **In/out marks on source:** `Footage::using_inout`, `in`, `out` fields. Set via I/O keys, focus-routed via `focusfilter.cpp`
- **In/out marks on timeline:** `Sequence::workarea_in/out`. Same I/O keys on sequence viewer
- **Insert mode:** `Timeline::paste(true)` (`timeline_clipboard.cpp:152-155`) — splits at playhead, ripples right, inserts clips
- **Overwrite mode:** `Timeline::paste(false)` — deletes in range, places clips

**What's missing (the actual work):**
1. **Drag respects footage in/out** — `TimelineWidget::dropEvent()` ignores `Footage::in/out`. Need to clip the dragged media to the marked range instead of full duration. Integration: `timelinewidget.cpp:294-400`
2. **"Insert from source" action** — single keyboard shortcut that takes footage viewer's in/out range, inserts at sequence playhead (or sequence in-point), ripples timeline. New method in `OliveGlobal`, wires to existing `Timeline::paste(true)` logic
3. **"Overwrite from source" action** — same but overwrites instead of rippling. Wires to `Timeline::paste(false)`
4. **Drag metadata** — `Viewer::initiate_drag()` (`viewer.cpp:550`) needs to include in/out marks in MIME data so drop handler can read them

**Not needed:** No new panels, no new viewer classes, no in/out mark infrastructure. It's ~200-300 lines of wiring between existing systems.

### F-8: Layout presets

**Depends on:** All UI features that presets reference.

Panel system already has `LoadLayoutState()`/`SaveLayoutState()`. Presets = named QByteArray blobs + floating window state. Mostly wiring, not architecture.

### F-9: .amb project format

**Depends on:** All features (captures all new data).

**Open question:** Does .ove actually need replacing? The existing XML schema handles:
- New effects: auto-discovered by name, `VoidEffect` for unknown
- New fields: matched by ID, unknown fields ignored gracefully
- Custom data: `custom_load()`/`save()` override
- Keyframe evolution: type field supports new interpolation types

The main thing .ove can't represent well: GPU shader effect parameters with non-standard types (textures, samplers). But 2.0 GPU effects use the same `EffectField` types as existing effects. May not need .amb at all for 2.0 — revisit when a concrete limitation surfaces.

### P-1: Rendering pipeline optimizations

Grouped by actual impact based on code audit.

**High impact:**

- **SI-1: RHI pipeline caching** — See SI-1 above. Biggest bang for buck. Single integration point (`rhi_blit()`). Eliminates ~80% of per-frame GPU resource allocations.

- **Half-res readback during scrub** — Currently full-res always (`renderthread.cpp:305-323`). Could create a half-res intermediate texture and readback that instead during scrub. Cuts readback bandwidth 4× (8.3 MB → 2.1 MB for 1080p, 35 MB → 8.8 MB for 4K). Integration: conditional in `paint()` based on `scrubbing` flag already available.

**Medium impact:**

- **Viewer overlays via QRhi** — Replace sibling QPainter QWidget with RHI vertex drawing. Fixes Vulkan compositing workaround, removes the extra compositing layer. But it means rewriting `draw_title_safe_area()`, `draw_guides()`, `draw_gizmos()` from QPainter to QRhi vertex buffers. Medium effort (~300-400 lines). Could be deferred if the sibling workaround holds.

- **Pre-allocated YUV staging buffers** — Replace per-frame `QByteArray` construction in `Clip::Retrieve()` with persistent pre-allocated buffers. Eliminates heap churn, not the copy itself (copy is mandatory, see YUV upload analysis). Low effort (~20 lines).

**Low impact / dropped:**

- **Zero-copy YUV upload** — Not feasible with Qt 6.8 RHI API. `QRhiTextureSubresourceUploadDescription` requires QByteArray. The copy overhead (90 MB/s at 30fps) is negligible anyway. **Dropped.**

- **GPU-only pipeline** — Achievable for 2.0 as an opt-in mode. Once F-1 ships GPU replacements for common Frei0r effects, Frei0r becomes a runtime toggle (off by default). With Frei0r disabled, `NeedsCpuRgba()` is never true → pure GPU path. Temporal effects (deshake, stabilization) require re-enabling the toggle. See F-1.

### P-2: Audio pipeline cleanup

**Bug fix (do first):**
- `audio_ibuffer` data race — AudioSenderThread reads at `audio.cpp:234, 247, 272` without `audio_write_lock`. Add lock around `send_audio_to_output()` buffer reads. Straightforward fix.

**Optimizations:**
- Pre-allocate `staging_buffer_` in AudioSenderThread constructor (currently `resize()` per scrub grain at `audio.cpp:222`).
- Scrub debouncing: skip grain decode if new seek within <16ms. Compare `audio_scrub_id` timestamps.

**ROADMAP correction:** The shared state atomics issue is already fixed. `audio_ibuffer_frame`, `audio_ibuffer_read`, `audio_reset_` are all `std::atomic<>`.

### F-15: Olive reference cleanup

Remove all remaining references to "Olive" in the codebase. The rebrand to Amber shipped in 1.2.0 (commit 420f5170e) but internal references remain scattered across CMake, source files, and config paths.

**Known locations (non-exhaustive, needs full audit):**
- CMakeLists.txt — variable names (`OLIVE_TARGET`, `OLIVE_SOURCES`, `OLIVE_RESOURCES`, etc.)
- Internal identifiers — class names, namespace references, comments
- Config paths — Qt settings org/app names if still referencing Olive
- Translation keys — any remaining `olive_` prefixed strings
- Log messages, error strings, window titles

**Approach:** Grep-based audit for `olive`/`Olive`/`OLIVE` across `src/` and `CMakeLists.txt` files. Rename in bulk. Test build. No functional change — purely cosmetic/consistency.

**Risk:** Low. No API or format changes. Internal-only identifiers. Could land in 1.4 or 2.0.

### F-16: Core/UI decoupling

Clean separation between engine logic and Qt Widgets. Not shipping separate libs or a CLI — just making the code split clean enough that one *could* build a headless frontend if they wanted. No contractualized API, no library packaging.

**Current state (from audit):**
- `core/` and `engine/` — already clean, zero UI imports
- `rendering/` — mostly clean, minor coupling to `amber::ActiveSequence`
- `effects/` — major coupling. `EffectField` subclasses inherit QWidget. `effect.cpp` calls QMessageBox/QFileDialog with `amber::MainWindow` as parent
- `global/` — god object. `OliveGlobal` mixes project lifecycle (save/load) with UI orchestration (dialogs, panel management)

**Incremental plan:**
1. Decouple `rendering/` — pass `Sequence*` explicitly instead of `amber::ActiveSequence`
2. Split `EffectField` into data model (value + keyframes + id) and widget (slider/picker/combo) — ~20 classes to refactor. The big one.
3. Split `OliveGlobal` into project I/O (headless-safe) + UI orchestration (dialogs, panels)
4. Keep monolithic `add_executable` — no CMake library targets needed

**Benefit:** Testable effects without UI, saner architecture, and if someone crazy wants to build their own frontend they can.

**Risk:** High effort on step 2 (effect field split). Can be done incrementally.

### T-1: Headless effect integration tests

**Depends on:** F-16 (Core/UI decoupling — done)

**Goal:** Regression safety net for the GPU-only pipeline push. Every effect that ships in 2.0 must have at least one headless test proving it produces the expected output. When F-1 ports Frei0r effects to GPU shaders, these tests prove the port is correct. When blur/chroma key/color correction get refactored, breakage shows up in `ctest`, not visually 3 weeks later.

**Approach:**
- Generate test inputs at test time with ffmpeg/ImageMagick: `ffmpeg -f lavfi -i color=black:s=64x64:d=1` for video, `magick -size 64x64 xc:black black.png` for images. No binary artifacts committed.
- Feed through `compose_sequence()` headless (QRhi OpenGLES2 null-surface, already proven in `test_rendering.cpp`).
- Apply effect, readback result, assert pixel values / clip duration / audio samples.

**Coverage — all non-deprecated effects (40+ tests):**

*Internal effects (12 — skip Frei0r, Mask disabled, VST):*
- **Transform:** translate → pixel shifted, scale 2× → dimensions doubled, rotation 180° → flipped
- **Text/RichText:** render text → non-black pixels in expected region
- **Solid:** color output matches configured color
- **Volume:** gain 0 → silence, gain 1 → passthrough
- **Pan:** hard left → right channel silent
- **Tone:** generates audio signal (non-zero samples)
- **Noise (audio):** generates non-silence
- **Shake:** output differs from input (non-deterministic, just verify it moved)
- **Timecode:** renders non-black pixels (burn-in text)
- **CornerPin:** known distortion → expected pixel shift
- **FillLeftRight:** channel duplication verified
- **Subtitle:** active entry → rendered text, no entry → transparent

*XML shader effects (29 — all must pass):*
- **Deterministic pixel tests:** invert (black→white), flip (horizontal/vertical pixel check), crop (border pixels transparent), posterize (reduced color count), chromakey/lumakey (known key color → transparent), colorsel (isolate channel), huesatbri (saturation 0 → grayscale), colorcorrection (known shift), emboss/toonify (output ≠ input), noise (output ≠ input), vignette (center brighter than corners), pixelate (adjacent pixels equal within block)
- **Blur family regression (critical — recently refactored):** boxblur, gaussianblur, directionalblur, radialblur — radius 0 = passthrough, radius N = output differs from input, blur(solid color) = same solid color
- **Distortion effects:** bulge, fisheye, sphere, swirl, ripple, wave, chromaticaberration, tile — output ≠ input, known center pixel preserved for symmetric distortions
- **Passthrough sanity:** every effect at default params on a known input → no crash, valid RGBA output

**Tolerance:** Pixel assertions use ±2 tolerance (GPU float rounding). Exact match only for trivial cases (solid color, full transparency).

**Test infra:**
- Single test binary `amber-test-effects-gpu` (or extend existing `amber-test-effects`)
- Helper: `create_test_sequence(width, height, fps)` → `Sequence*` with one clip
- Helper: `apply_effect(clip, effect_meta, params...)` → attach + configure
- Helper: `render_and_readback(sequence, frame)` → `QByteArray` RGBA pixels
- Helper: `assert_pixel(pixels, x, y, expected_rgba, tolerance)`

**Frei0r:** Out of scope — deprecated in 2.0, CPU-only, not part of the GPU-only push.
